# AdHunter 来宾机行为分析报告

日期：`2026-04-23`
实验虚机：`AdHunterLab`

## 分析范围

本报告总结了在隔离的 Hyper-V 虚机内运行下载得到的客户端后，客体系统侧观察到的执行行为。重点关注：

- 持久化方式
- 内核态与用户态模块
- 云端下发子模块的证据

## 高置信度结论

### 1. 主进程运行后会随机化进程名

磁盘上的主程序路径始终是：

- `C:\Analysis\adhunter\17629075264\17629075264\AdHunter.exe`

但启动后，进程名会变成随机值，例如：

- `puxgwllk.exe`
- `fuhmceou.exe`

观察到的运行会话：

- 通过 `PowerShell Direct` 启动时在 `Session 0`
- 通过桌面自启路径启动时在 `Session 1`

这说明它没有改动原始文件路径，但会对运行时进程身份做混淆。

### 2. 每次运行都会安装一个随机名的内核驱动服务

已观察到的驱动服务名：

- `k2ayuj25q`
- `l9qtsnk`

对应驱动落地路径：

- `C:\Windows\o8w6m5\k2ayuj25q.sys`
- `C:\Windows\jueikzd56\l9qtsnk.sys`

系统事件中观察到：

- `Service Control Manager 7045`：安装了内核模式驱动服务
- `Microsoft-Windows-FilterManager 6`：筛选器成功加载
- `AFD 16001`：出现 TDI 筛选器警告

`fltmc` 已确认当前活动的微型筛选驱动：

- 筛选器名：`l9qtsnk`
- 高度：`371912`

拷回宿主机分析后，驱动本体信息如下：

- 原始文件名：`AdDriver.sys`
- 产品名：`振创网络净化中心`
- SHA256：`D322ECFA2B37F23A7510957F46FCAD5879F5A5A46A0CBD5B797E960954341EC6`

也就是说，服务名和落地目录是随机的，但驱动本体是有签名的正式驱动。

## 用户态模块链

### 3. 主进程会加载一套隐藏的 ControlWindow 模块

从运行中的随机名主进程枚举模块时，确认加载了：

- `C:\Windows\zz6c4gordd\Module\ControlWindow\ControlWindow.dll`

从客体系统完整拷回的目录内容如下：

- `ControlWindow.dll`
- `cwapi_32.dll`
- `hkthunk.dll`
- `hkthunk64.dll`
- `remote.exe`
- `GLLA.exe`
- `ControlWindow.dll.__config.ini`

以上文件全部未签名。

哈希比对已经确认：

- `ControlWindow.dll` 就是之前落地的 `vqvam_80p_x64.dll`
- `cwapi_32.dll` 就是之前落地的 `vqvam_80p_x86.dll`
- `hkthunk.dll` 就是 `fora9_6e7oc1i`
- `hkthunk64.dll` 就是 `6fora9_6e7oc1i`

这说明先前看起来像“随机 DLL / 随机二进制块”的文件，其实是一整套窗口控制与注入模块。

### 4. ControlWindow 明显是注入 / Hook 子系统

`ControlWindow.dll` 与 `cwapi_32.dll` 中的高信号字符串包括：

- `Hook_SetWindowPos`
- `InstallHook`
- `UninstallHook`
- `LoadInjectDllThunk`
- `ControlWindow.dll`
- `ControlWindowApi.dll`
- `CW!Intercept`
- `CW!Hook_CoCreateInstanceEx`
- `CW!SendAijDllPath`
- `ADH!CW::CopyDll`

两个 `hkthunk*.dll` 中还保留了 PDB 路径：

- `d:\project\zc\adblock\lmp\inject_thunk\objchk_win7_x86\i386\thunk.pdb`
- `d:\project\zc\adblock\lmp\inject_thunk\objchk_win7_amd64\amd64\thunk.pdb`

这基本可以确认，它的“窗口控制”能力并不只是枚举顶层窗口，而是依赖 DLL 注入和 Hook 辅助模块。

### 5. `remote.exe` 暴露了模块路由与远程加载痕迹

`ControlWindow` 目录中的 `remote.exe` 包含这些关键字符串：

- `http://1.adhunter.cn/index.php/client_get_data/get_ip_agent_name`
- `http://2.adhunter.cn/index.php/client_get_data/get_ip_agent_name`
- `http://3.adhunter.cn/index.php/client_get_data/get_ip_agent_name`
- `http://z.wb009.com/nav2/%s/remote.txt`
- `www.adhunter.cn`
- `LoadLibraryA`
- `CreateRemoteThread`
- `RunRemoteExe.exe`
- `ZCRemote`

这说明它很可能承担了第二阶段功能：

- 根据 IP 或环境解析模块路由
- 拉取远程配置或指令
- 在目标进程里远程加载用户态组件

## 云端下发子模块的直接证据

### 6. 客户端确实会下载与微信场景相关的清理包

在客体系统的 Internet Cache 中发现了：

- `C:\Users\Administrator\AppData\Local\Microsoft\Windows\INetCache\IE\N0DY9JEA\WxPlugin[1].zip`

压缩包内文件：

- `WxPlugin.bat`

该批处理脚本不是普通配置文件，而是带有明确场景目标的清理脚本，会执行：

- `net stop wxpluginad`
- 删除 `wxpluginad.sys`
- 删除 `C:\windows\syswow64\drivers\*.dat`
- 处理 `C:\windows\syswow64\Wxplugin.sys`
- 给 `P2PSyncService.exe` 设置 IFEO

它还会检查多种网吧环境下的管理标记：

- `iCafe8`
- `EYOOCLIENTSTATUS`
- `NXD`
- `nds`

这说明广告猎手不是只靠一个总控客户端工作，它还能从云端下发针对具体环境的专项修复包。

### 7. `cw.dat` 很可能是加密或打包后的本地状态 / 规则数据

客户端还创建了：

- `C:\Windows\cw.dat`

属性如下：

- 大小：`15288`
- SHA256：`8475A94457DDDCD035B03D76022AC5A6F3001E6B9E0BFD3A4738935612CFE2DB`

该文件没有明显明文字符串，文件头呈现高熵二进制数据，更像是：

- 加密后的本地状态
- 压缩后的规则缓存
- 或 ControlWindow 子系统使用的二进制配置块

## 架构层面的含义

现有证据支持下面这条执行链：

1. `AdHunter.exe` 启动后先混淆自己的运行时进程名。
2. 它部署一个带签名、但服务名和目录随机化的微型筛选驱动。
3. 它在 `C:\Windows\<随机目录>` 下布置隐藏的 `ControlWindow` 模块目录。
4. 主进程加载 `ControlWindow.dll`，并使用 `hkthunk` 辅助 DLL 做注入 / Hook。
5. 它能从云端拉取专项修复包，例如 `WxPlugin.bat`。
6. 它还带有 `remote.exe` 这类远程模块辅助程序，用于解析和加载后续行为模块。

所以它远不只是“下载一份规则文本”。目前更接近一个模块化的 Windows 控制代理，具备：

- 已签名的内核执行层
- 隐藏的用户态窗口 / 注入模块
- 云端专项包投放能力
- 面向网吧环境的修复脚本
- 通过随机命名降低静态 IOC 稳定性的能力
