# 广告猎手开机启动长跟踪报告

时间：2026-04-23

## 结论

这次在虚拟机里做的是“真实桌面登录后自动启动”的长时跟踪，不是手工双击运行。结果已经可以把它的启动动作串起来：

1. 开机登录后，`AdHunter.exe` 被启动脚本拉起，但运行中的进程名会立刻伪装成随机名。
2. 启动瞬间会安装一个随机命名的内核驱动服务，并把驱动文件落到 `C:\Windows\<随机目录>\<随机名>.sys`。
3. 紧接着会在 `C:\Windows\<随机目录>\Module\ControlWindow\` 下部署一整套窗口控制/注入模块。
4. 主进程会加载 `ControlWindow.dll`，并持续写入 `AdhDriverLog.txt`。
5. 同一时间还会往当前用户的 `INetCache` 里下发一个专项 ZIP 包 `WxPlugin[1].zip`，内容是专门处理网吧环境里微信插件广告/同步组件的批处理。

这已经足够说明它不是“单纯拉一份规则文本”，而是“开机即装载驱动 + 本地模块解包 + 云端专项包下发”的架构。

- 或通过其他辅助模块完成；
- 或先拉包后快速断开，没有在 5 秒采样窗口里稳定留下。

但是结合 DNS 命中、`INetCache` 的 `WxPlugin[1].zip` 实际落地，以及模块目录在同一时间生成，已经可以确认“启动后联网下发”这条链是真实存在的。

## 对“网吧无盘开机还原”场景的判断

如果放回你一开始说的无盘网吧场景，这个客户端的设计非常像：

1. 客户端通过启动链尽早运行。
2. 开机后先把驱动执行层和窗口控制模块装起来。
3. 立刻联网获取当前环境对应的清理包/规则包。
4. 把专项模块落到本地随机目录或缓存目录。
5. 再由驱动层和用户态窗口控制模块一起对计费、无盘、微信插件、广告窗口做处理。

也就是说，它解决“无盘开机还原”问题的核心思路不是把所有规则固化在本地，而是：

- 本地常驻执行层固定；
- 具体规则和专项修复包走云端下发；
- 每次开机重新取当前策略。

- 有随机化进程名。

- 有驱动层多模块初始化。
- 有持续本地日志。
- 有启动后几秒内的专项 ZIP 下发。

2. 抓 `ControlWindow` 模块目录是本地解包还是远端下载更新。
3. 在后台只加一条你自己的实验规则，再比对 `Rule Count`、`cw.dat` 和模块目录变化。
